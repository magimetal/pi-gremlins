# Pi 0.75.3 -> 0.77.0 Compatibility Update Plan

## Objective

Update `pi-gremlins` for compatibility with Pi `0.77.0` from `0.75.3`, applying only relevant changes. The changelog review found no explicit breaking changes, so this is a dependency-alignment and regression-hardening update rather than a runtime rewrite.

## Assumptions

- Pi `0.77.0` packages are available under the existing `@earendil-works/*` namespaces.
- `peerDependencies` should remain wildcard to match Pi package-extension guidance and avoid forcing host installs.
- The existing `pi.registerTool(tool as any)` workaround remains necessary unless typecheck against `0.77.0` proves upstream typings now avoid `TS2589`.
- No PRD/ADR is required because this is compatibility maintenance, not a product or architectural decision.

## No-Change Rationale for Irrelevant Changelog Items

- No explicit `Breaking Changes` entries were identified for `>0.75.3` through `0.77.0`.
- New streaming-aware extension input event (`pi.on("input")`, `event.streamingBehavior`) does not require code changes because `pi-gremlins` does not register an `input` handler.
- `getAllTools` prompt-guideline fixes do not apply because `pi-gremlins` does not call `getAllTools`.
- `--exclude-tools` support is host/user behavior. No extension code is required, beyond optional README troubleshooting notes.
- TUI changes appear non-breaking; verify overlays and inline rendering manually instead of preemptive refactors.

## Phase 2 — Verify and Harden SDK Session Lifecycle Compatibility

### Task 2.1 — Regression-test child session disposal and abort paths

**What**

**References**

- `extensions/pi-gremlins/gremlins/gremlin-runner.ts`

  - `extensions/pi-gremlins/test/gremlins/gremlin-session-factory.test.js`
  - `extensions/pi-gremlins/test/side-chat/side-chat-command.test.js`
  - `extensions/pi-gremlins/test/side-chat/side-chat-session-factory.test.js`

**Expected edits**

- Prefer no runtime code changes if current tests pass.
- Add or adjust tests only if `0.77.0` changes observable behavior around `abort()`, `dispose()`, subscription cleanup, or rejected prompts.
- If a test exposes a bug, keep fixes local to the session lifecycle code paths above.

**Acceptance criteria**

- Parent abort still marks active gremlins canceled and disposes child sessions.
- `session_shutdown` clears discovery caches, active gremlin sessions, gremlin transcripts, side-chat subscriptions, and overlays.
- Side-chat reset/session switching disposes the prior child session without leaking subscriptions.
- No unhandled rejection occurs when abort/dispose races with prompt completion.

**Guardrails**

- Do not alter gremlin prompt isolation or side-chat resource-loader isolation.
- Do not add parent resources, prompts, themes, AGENTS files, or primary-agent material to child sessions.
- Do not catch and suppress lifecycle errors unless the current behavior is already intentionally best-effort, such as already-disposed side-chat sessions.

## Phase 3 — Validate Streaming and Repeated Prompt Behavior

### Task 3.1 — Add side-chat repeated-prompt regression coverage if missing

**What**

Verify behavior when `/gremlins:chat` or `/gremlins:tangent` receives a second prompt while the active child session is still streaming. Research indicates this risk predates `0.77.0`, but the `0.77.0` streaming-aware changes make it important to lock down expected behavior.

**References**

- `extensions/pi-gremlins/side-chat/side-chat-command.ts`
- `extensions/pi-gremlins/side-chat/side-chat-session-factory.ts`
- `extensions/pi-gremlins/test/side-chat/side-chat-command.test.js`
- `extensions/pi-gremlins/test/side-chat/side-chat-transcript-state.test.js`

**Expected edits**

Choose the smallest safe option based on observed test behavior:

1. If SDK `session.prompt()` rejects while streaming and current code already surfaces the error cleanly, add/keep a regression test asserting:
   - user message appears in overlay transcript,
   - prompt failure is rendered via `appendSideChatError`,
   - UI notification says `Side-chat prompt failed: ...`,
   - pending question state does not corrupt the next successful exchange.
2. If rejection leaves stale pending questions or transcript persistence mismatch, fix pending-question rollback in `side-chat-command.ts` for failed prompts.
3. Do not adopt `pi.on("input")` or `streamingBehavior` unless a future product change intentionally routes side-chat through input events.

**Acceptance criteria**

- Repeated prompt during streaming has deterministic behavior: either safely rejected with visible error and clean state, or queued intentionally with tests.
- Chat and tangent modes keep independent pending-question queues.
- Persisted side-chat exchanges pair each completed answer with the correct question.

**Guardrails**

- Do not implement broad side-chat UX redesign.
- Do not switch from `prompt()` to `steer()` or `followUp()` without confirming the SDK contract and updating tests accordingly.
- Do not use `streamingBehavior` outside input-event handlers.

**Verification**

```bash
npm test -- extensions/pi-gremlins/test/side-chat/side-chat-command.test.js
npm test -- extensions/pi-gremlins/test/side-chat/side-chat-persistence.test.js
```

## Final Verification Sequence

Run after implementation edits are complete:

```bash
npm install
npm run typecheck
npm test
npm run check
```

Manual smoke checks in Pi `0.77.0` host:

1. Install/reinstall the local package into Pi.
2. Start a session and confirm extension registration succeeds.
3. Invoke `pi-gremlins` with one simple gremlin and confirm inline collapsed/expanded rendering works.
4. Abort an active gremlin and confirm canceled status plus cleanup.
5. Use `/gremlins:steer <G-id> <message>` during an active gremlin and confirm queued/rejected activity is visible.
6. Open `/gremlins:chat`, submit a prompt, and confirm overlay transcript renders.
7. Submit a second side-chat prompt while the prior response is streaming and confirm the expected tested behavior.
8. Send SIGTERM/SIGHUP or otherwise end the host session and confirm session shutdown cleanup causes no visible errors.
